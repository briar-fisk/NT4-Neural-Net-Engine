\doxysubsection{Detailed Description}
This is a handshake based text interface for the engine. 

File-\/\+Based Command Control System for Asynchronous Standalone Neural-\/\+Net Engine Control\+:

Text-\/\+Based Command Execution\+:
\begin{DoxyItemize}
\item Commands are represented as text strings that specify actions or operations to be performed by the system or process.
\item Each command is associated with a specific functionality or operation within the system.
\end{DoxyItemize}

File I/O\+:
\begin{DoxyItemize}
\item Input and output operations are performed using files as the medium of communication.
\item Commands are read from input files, and outputs, returns, and status indicators are written to output files, though some commands do output to the console.
\item This method facilitates asynchronous communication between systems, as they can read from and write to files independently.
\end{DoxyItemize}

Handshake Protocol\+:
\begin{DoxyItemize}
\item A handshake protocol is used to establish communication or synchronize actions between systems.
\item In this context, a flag file serves as a mechanism for signaling the readiness or completion of certain operations, namely that the user has written a command sequence to \textquotesingle{}Control\+\_\+\+Panel.\+ssv\textquotesingle{} that is ready to be interpreted..
\item The content of the flag file is used to indicate the status of the control panel, allowing the system to either wait or act accordingly.
\end{DoxyItemize}

Flag File\+:
\begin{DoxyItemize}
\item A flag file is a small file used to signal the status or state of the system.
\item It typically contains simple data or metadata, such as a single value or indicator, to convey information. In the \doxylink{namespace_n_t4}{NT4} text interface it is a boolean value.
\item Flag files are used as synchronization primitives to coordinate activities between systems.
\end{DoxyItemize}

Control Panel\+:
\begin{DoxyItemize}
\item The control panel represents a set of commands or instructions that can be executed by the system or process.
\item It is stored in a separate file (e.\+g., Control\+\_\+\+Panel.\+ssv), which contains a sequence of commands or directives to be processed.
\end{DoxyItemize}

Interpretation and Execution\+:
\begin{DoxyItemize}
\item The system interprets commands read from the control file (e.\+g., Control\+\_\+\+Panel.\+ssv) and executes them sequentially.
\item Command execution may involve performing operations on data, manipulating the state of the system, or interacting with external resources.
\end{DoxyItemize}

Status Checking and Control Flow\+:
\begin{DoxyItemize}

\item Whitespace is weird when doing text input through set\+\_\+\+Input! It reads from the text file using "{}file $>$$>$ tmp\+\_\+\+Variable"{}, then this variable is concatenated to the others, "{}full\+\_\+\+Variable = full\+\_\+\+Variable + "{} "{} + tmp\+\_\+\+Variable. This full string is then read in as input and the network evaluates each character including spaces, even though whitespace was ignored in tokenization. So be aware these two behaviours stack to form a strange and peculiar action.

Additional Notes\+:
\begin{DoxyItemize}
\item Use characters or numbers that you know won\textquotesingle{}t appear in the dataset to create a "{}greenscreen"{} over your input for the network. This is due to the NULLCAN, where a node that isn\textquotesingle{}t found is set to NULL on the CAN scaffold, and subsequently is skipped during charging as there is nothing to charge. Alternatively just use the query\+\_\+\+Given\+\_\+\+Legs command.
\item Whitespace is weird when doing text input through set\+\_\+\+Input! It reads from the text file using "{}file $>$$>$ tmp\+\_\+\+Variable"{}, then this variable is concatenated to the others, "{}full\+\_\+\+Variable = full\+\_\+\+Variable + "{} "{} + tmp\+\_\+\+Variable. If the next token is the \textquotesingle{}/end/\textquotesingle{} tag it does not append whitespace to the token, so \textquotesingle{}set\+\_\+\+Input 0 10 /end/\textquotesingle{} would only input \textquotesingle{}10\textquotesingle{} not \textquotesingle{}10 \textquotesingle{}. This full string is then read in as input and the network evaluates each character including spaces, even though whitespace was ignored in tokenization. This means that if you are trying to search for multiple tokens in a single string you need to be aware that spaces between the words may cause false positive associations due to the space or other whitespace. The greenscreen masking technique detailed in set\+\_\+\+Input can be used to handle this behavior through input formatting only. So be aware these two behaviours stack to form a strange and peculiar action.

This function works by taking this set of \textquotesingle{}legs\textquotesingle{} which is a boolean array. It then takes every input of the construct and puts them in a pool of unorganized nodes. It then steps through the boolean array checking whether that step needs to be charged or not. If it needs charged the program then steps through every node in the pool having it charge any axons on the current index, which would be the axon\+\_\+hillock to the node.

So if we are on the second leg in the array then we would step through each node, go check the axon\+\_\+hillock population count, and if any are found we charge them into the charging buffers.

This means you will search for the set of inputs you give, not in order but individually, on the legs you specify.
